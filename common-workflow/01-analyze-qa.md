# Step 01 - Analyze Requirement and Clarify with User

## Goal

在開始任何規劃、切分支、撰寫程式碼之前，先完整分析需求，主動找出不清楚、矛盾、缺漏、風險與依賴，並整理成可確認的需求摘要，回覆給使用者確認。
只有在使用者確認理解正確後，才可進入 Step 02。

---

## Core Responsibilities

1. 解析使用者需求，判斷這次是：
   - 新專案建立
   - 既有專案功能開發
   - 小幅修正 / bugfix
   - 架構調整 / 重構
   - 純文件或規劃需求

2. 主動釐清：
   - 功能目標
   - 使用者情境
   - 輸入 / 輸出
   - UI/UX 預期
   - API / DB / 權限 / 邏輯規則
   - 邊界條件
   - 錯誤處理
   - 驗收標準
   - 是否有既有檔案 / 模組 / 架構要延續
   - 是否有指定技術、套件、命名、風格限制

3. 如果資訊不足，不可直接開始寫 code，必須先提出精準問題。

4. 分析後要輸出一份「我理解的需求摘要」，讓使用者確認。

---

## Execution Rules

- 不可跳過需求分析直接進入實作。
- 不可假設未提供的重要規格。
- 若需求模糊，必須優先提問。
- 問題要具體，不要問太空泛的問題。
- 若需求已足夠，也要主動整理成條列摘要給使用者確認。
- 必須明確詢問：「以上理解是否正確？如果沒問題，我將進入 Step 02 規劃與切分實作步驟。」

---

## Required Output Format

請以以下格式回覆使用者：

### 1. Requirement Summary

- 本次目標：
- 預期結果：
- 影響範圍：
- 不在此次範圍內：

### 2. Functional Details

- 使用者操作流程：
- 資料流 / 狀態流：
- 需要處理的邏輯：
- 例外 / 邊界情況：

### 3. Risks / Missing Information

- 尚需確認：
- 潛在風險：
- 依賴項目：

### 4. Confirmation

- 請使用者確認目前理解是否正確
- 若確認無誤，再進入 Step 02

---

## Exit Criteria

只有在以下條件都成立時才能進入 Step 02：

- 需求已整理完成
- 關鍵不確定點已釐清，或已明確標示假設
- 使用者已確認可以往下走

若未確認，停留在 Step 01 持續問答釐清。
